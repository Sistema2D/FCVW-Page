# Data, Persistence, and Migration

Methodological document to guide how the application stores, reads, protects, migrates, exports, imports, and discards data.

This file should be consulted before making changes to databases, local files, JSON, cache, logs, vaults, chat history, settings, import, export, backup, migration, or retention.

## Objective

Ensure that user data and versioned data are handled in a secure, predictable, auditable manner that is compatible with the application's versioning.

## Principles

- User data must not be versioned.
- Versioned data must be separated from runtime-generated data.
- Every critical write must consider backup, atomic writing, or rollback.
- Migrations must be idempotent whenever possible.
- Corrupted data must be handled without silent loss.
- Sensitive data must not be recorded in logs.
- Format changes must be registered in the plan and changelog.

## Data Classification

All data handled by the application must be classified into one of the four categories below. The classification determines where the data can be stored, whether it can be versioned, how it should be protected, and what happens when it is discarded.

The categories are: **versioned** (part of the repository), **user** (generated by usage), **temporary** (recreatable without loss), and **sensitive** (requires additional protection). A data item can belong to more than one category.

### Versioned Data

They are part of the repository and can be distributed with the application.

Examples:

- offline catalogs;
- templates;
- assets;
- documentation;
- default settings without secrets.

### User Data

Generated or modified by using the application.

Examples:

- personal settings;
- chat history;
- local notes;
- imported files;
- local database;
- usage logs.

### Temporary Data

Can be recreated without significant functional loss.

Examples:

- cache;
- intermediate files;
- builds;
- generated indexes;
- transient results.

### Sensitive Data

Requires additional protection.

Examples:

- tokens;
- keys;
- credentials;
- personal data;
- health data;
- private documents;
- confidential chat content.

## Data Versioning Rules

- User data must remain out of Git.
- Sensitive data must never be versioned.
- `.gitignore` must cover logs, builds, caches, local vaults, and private configuration files.
- Example files must use placeholders.
- Changes to versioned data require a changelog.

## Persistence in Files

When the application uses local files:

- prefer readable formats when appropriate, such as Markdown, JSON, YAML, or TOML;
- validate content before overwriting an existing file;
- use atomic writing for important data;
- create a backup before a critical rewrite;
- handle file absence with secure creation;
- handle corrupted files with documented fallback;
- record schema changes.

## Persistence in Database

When the application uses a database:

- document engine, version, and storage location;
- create a migration strategy;
- record creation and update scripts;
- define backup and restore procedures;
- separate application data from user data when possible;
- test migration with previous data;
- record rollback procedures when possible.

If the project does not use a database, record this explicitly. The future adoption of a database must be treated as a change of scope and architecture, requiring a plan, architectural decision, data analysis, security, testing, and changelog.

## Data Migration

Every migration must contain:

- source version;
- target version;
- affected data;
- transformation applied;
- backup before migration;
- behavior in case of failure;
- rollback possibility;
- post-migration validation.

## Rules for Import

- Every imported file must be treated as untrusted.
- Validate extension, size, and content.
- Block dangerous paths.
- Do not execute imported content.
- Preserve raw source when necessary.
- Record origin and date when relevant.
- Handle duplications.

## Rules for Export

- Inform the user what will be exported.
- Do not include secrets by default.
- Allow readable format when applicable.
- Record the schema version in the exported file.
- Document compatibility limitations.

## Retention and Discard

Define, when applicable:

- how long logs are kept;
- when cache can be cleared;
- how to delete user data;
- how to confirm destructive actions;
- if deletion is reversible;
- if there is a backup before deletion.

## Schema and Compatibility

Every persisted format must have, when applicable:

- schema name;
- schema version;
- mandatory fields;
- optional fields;
- default values;
- validation rules;
- backward compatibility.

## Database Schema Creation and Update Protocols

To prevent data drift and ensure that databases created by the user are always in sync with the active application version, the following operational protocols must be followed:

### 1. Database Creation Blueprint
* **Pristine Schema Mapping:** Every database table must have an associated SQL DDL blueprint recorded in `governance/TEMPLATE_DATA_SCHEMA.md` or a dedicated markdown record in `/data/schemas/`.
* **Standard SQLite Engine:** Since SQLite is the recommended local persistence database, all schema DDLs must follow standard ANSI SQL optimized for SQLite 3.x.
* **Schema Version Tracking:** Every database must contain a core metadata table named `schema_metadata` or utilize the SQLite `user_version` PRAGMA to record the current active schema version:
  ```sql
  CREATE TABLE IF NOT EXISTS schema_metadata (
      version INTEGER PRIMARY KEY,
      applied_at TEXT NOT NULL,
      description TEXT NOT NULL
  );
  ```

### 2. Automated Schema Update Engine
To automate schema upgrades safely without relying on external system script dependencies, the application's startup lifecycle must implement a light **Migration Runner** following these rules:
1. **Startup Check:** On application startup, the database manager queries `schema_metadata` (or PRAGMA `user_version`) to identify the current database schema version.
2. **Sequential Upgrade (Idempotent Migrations):** The manager compares the database's version against the target schema version specified in `DATA.md`. If the database version is lower, the manager runs the sequential `.sql` migration scripts defined in `data/migrations/` (e.g., `V1__init.sql`, `V2__add_email_to_users.sql`) in a strict transaction block:
   ```sql
   BEGIN TRANSACTION;
   -- Apply DDL/DML upgrades here...
   INSERT INTO schema_metadata (version, applied_at, description) VALUES (2, datetime('now'), 'Add email field');
   COMMIT;
   ```
3. **Automatic Backup Before Rewrite:** Before executing any migration transaction, the manager must copy the current database file (e.g. `data/app.db`) to a temporary backup file (`data/app.db.bak`).
4. **Failure Recovery (Self-Healing Rollback):** If any query inside the migration transaction fails:
   * The transaction is rolled back immediately via `ROLLBACK TRANSACTION;`.
   * The manager restores the temporary backup file (`app.db.bak`) over the active database file.
   * The application halts execution and displays a clear error log, preventing silent corruption.

### 3. Verification & Maintenance Routine
* Whenever the developer or the AI adds a new field or table in `governance/TEMPLATE_DATA_SCHEMA.md`:
  1. A new migration `.sql` script must be added to `/data/migrations/`.
  2. The target schema version must be bumped in both `DATA.md` and `MANIFEST.md`.
  3. The visual directory tree in `FILESYSTEM.md` must be refreshed.

## Checklist for Data Changes

- [ ] Data type has been classified.
- [ ] Data should or should not be versioned.
- [ ] Storage location has been defined.
- [ ] There is a risk of sensitive data.
- [ ] There is a backup before a destructive change.
- [ ] There is a migration if the format changes.
- [ ] There is validation for corrupted files or invalid input.
- [ ] Old data reading has been tested.
- [ ] `.gitignore` covers private data.
- [ ] Changelog records data changes.

## Rule for Governance Wiki

When a governance `wiki/` exists, it must be treated as versioned documentation and must point to formal sources when recording learnings.

The governance wiki must not copy private data, sensitive logs, or chat content without anonymization and explicit justification.
